# Guia: Geracao de Questoes via LLM (provas)

Data: 2026-04-09

## 1. Objetivo

Orientar o uso operacional do endpoint de geracao de questoes via IA no modulo de provas.

Este guia e focado em:
- configuracao minima de ambiente;
- uso correto de `quantidade`;
- interpretacao dos erros mais comuns;
- boas praticas para frontend e backend.

## 2. Pre-requisitos

- prova existente em status `RASCUNHO`;
- usuario autenticado com `ROLE_PROFESSOR` (dono da prova) ou `ROLE_ADMIN`;
- chave de IA configurada:
  - `GOOGLE_API_KEY`; ou
  - `GEMINI_API_KEY`.

Exemplo em `.env`:

```env
GOOGLE_API_KEY=<sua-chave>
```

## 3. Endpoint

`POST /api/v1/provas/{id}/gerar-questoes`

Header:

```http
Authorization: Bearer <jwt>
Content-Type: application/json
```

Body minimo:

```json
{
  "tema": "Revolucao Francesa"
}
```

Body com quantidade explicita:

```json
{
  "tema": "Revolucao Francesa",
  "quantidade": 5
}
```

## 4. Regra pratica de quantidade

- `qntQuestoes` da prova define o total planejado.
- sem `quantidade`, a API gera automaticamente o restante planejado.
- o modo automatico aplica teto de 20 por chamada.
- se o restante for maior que 20, repita chamadas em lotes.
- se `quantidade` for maior que o restante, retorna 400.

## 5. Comportamento de validacao da IA

Antes de salvar, o backend valida:
- formato JSON esperado;
- quantidade retornada igual a solicitada;
- enunciados sem duplicacao;
- gabarito valido em A-D;
- exatamente 4 alternativas por questao;
- gabarito presente entre as alternativas.

Se falhar:
- tenta uma segunda chamada com prompt fallback;
- se falhar de novo, retorna 400 sem salvar parcialmente.

## 6. Limite de consumo

- limite atual: 5 chamadas por minuto por professor autenticado;
- excedeu limite: retorna 429;
- erro de elegibilidade (por exemplo, restante excedido) nao deve consumir cota.

## 7. Erros comuns e como agir

| HTTP | Cenario comum | Acao recomendada |
|---|---|---|
| 400 | prova publicada, restante excedido, payload invalido da IA | ajustar dados e reenviar |
| 401 | token ausente/invalido | renovar sessao e repetir |
| 403 | ownership invalido | revisar usuario logado |
| 404 | prova nao encontrada | validar id da prova |
| 429 | limite por minuto atingido | aguardar janela e tentar novamente |
| 503 | LLM indisponivel ou API key ausente | revisar ambiente/chave e tentar novamente |

## 8. Boas praticas

- sempre mostrar mensagem do backend (`message`) ao professor;
- em fluxo automatico, exibir progresso por lote quando restante > 20;
- nao repetir chamada em loop agressivo apos 429;
- manter revisao humana antes de publicar prova.

## 9. Checklist rapido de troubleshooting

1. prova esta em `RASCUNHO`?
2. usuario logado e dono da prova (ou admin)?
3. `qntQuestoes` da prova esta definida e > 0?
4. chave da IA esta carregada no ambiente ou `.env`?
5. houve excesso de chamadas no ultimo minuto?
6. retorno da IA foi invalidado pelas regras RN-06?
