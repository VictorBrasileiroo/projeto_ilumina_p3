# Guia: Uso de Access Token, Refresh Token e Claims

Data: 2026-04-05

## 1. Conceitos rapidos

- Access token: token curto para chamar APIs protegidas.
- Refresh token: token de renovacao de sessao.
- Claims: dados dentro do JWT para identificar contexto do usuario.

## 2. O que guardar no frontend

Minimo recomendado para sessao:
- access token
- refresh token
- userId
- professorId/alunoId (quando vier)
- roles

## 3. Como enviar token nas requisicoes

Header padrao:
```http
Authorization: Bearer <access-token>
```

Nao use refresh token para endpoint de negocio.

## 4. Quando usar refresh

Use refresh quando:
- access token expirar e API responder 401.

Passos:
1. Interceptar `401`.
2. Chamar `POST /api/v1/auth/refresh` com o refresh token.
3. Atualizar tokens armazenados.
4. Repetir request original uma vez.

## 5. Fluxo de frontend recomendado

## 5.1 Login

1. Usuario faz login.
2. API retorna access + refresh + metadados.
3. Front salva estado de sessao.
4. Front direciona para area autenticada.

## 5.2 Navegacao

1. Requests com access token.
2. Claims ajudam no bootstrap de tela (perfil/role).
3. Regras finais de autorizacao continuam no backend.

## 5.3 Expiracao

1. Access expirou -> 401.
2. Front chama refresh.
3. Recebe novos tokens.
4. Continua sem forcar relogin imediato.

## 5.4 Logout

1. Limpar access token.
2. Limpar refresh token.
3. Limpar store/context de usuario.

## 6. Claims uteis atuais

No access token:
- `sub` (email)
- `tokenType=access`
- `userId`
- `roles`
- `professorId` (quando existir)
- `alunoId` (quando existir)

No refresh token:
- `tokenType=refresh`
- `userId`

## 7. Exemplo de estrategia de interceptor (pseudo)

```ts
async function apiRequest(config) {
  const access = session.getAccessToken();
  config.headers.Authorization = `Bearer ${access}`;

  try {
    return await http(config);
  } catch (err) {
    if (err.status !== 401 || config.__retried) throw err;

    config.__retried = true;
    const refresh = session.getRefreshToken();
    const renewed = await http.post('/api/v1/auth/refresh', { refreshToken: refresh });

    session.setTokens(renewed.data.token, renewed.data.refreshToken);
    config.headers.Authorization = `Bearer ${renewed.data.token}`;

    return await http(config);
  }
}
```

## 8. Boas praticas

- Sempre validar `401` e `403` no front.
- Limitar retry de refresh para evitar loop infinito.
- Nao colocar regra de autorizacao sensivel so no frontend.
- Backend continua sendo a fonte de verdade para permissoes.
