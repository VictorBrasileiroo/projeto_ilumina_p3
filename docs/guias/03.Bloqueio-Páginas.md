# Guia: Cadastro de Professor com Login Imediato

Data: 2026-04-05

## 1. Como bloquear acesso a paginas?

Resposta curta: os dois lados importam.

- Frontend: controla navegacao e experiencia do usuario.
- Backend: e a protecao real dos dados e regras.

## 1.1 Bloqueio no frontend (guard de rota)

No front, voce cria guards para impedir que usuario sem token acesse rota privada.

Regras comuns:
- sem token -> redireciona para `/login`;
- com token mas sem role exigida -> redireciona para `/acesso-negado`.

Isso melhora UX, mas sozinho nao garante seguranca total (porque o cliente pode ser manipulado).

## 1.2 Bloqueio no backend (obrigatorio)

No backend, a seguranca precisa validar token e autorizacao em toda rota protegida.

No projeto atual isso ja existe com:
- `SecurityConfig` definindo rotas publicas e privadas;
- filtro JWT validando token;
- `@PreAuthorize` controlando papel (role) por endpoint.

Exemplo real do modulo professor:
- `POST /api/v1/professor` -> publico;
- `GET /api/v1/professor` -> admin;
- `GET/PUT /api/v1/professor/{id}` -> admin ou proprio professor;
- `PATCH /api/v1/professor/{id}/deactivate` -> admin.

## 2. Padrao recomendado para nao ter brecha

Use sempre este combo:
1. Front guard para navegacao.
2. Back guard para seguranca real.

Resumo pratico:
- o front barra visualmente;
- o back barra tecnicamente.

Se o usuario tentar chamar API protegida sem token ou sem permissao, o backend deve devolver `401` ou `403` mesmo que o front tenha falhado.
